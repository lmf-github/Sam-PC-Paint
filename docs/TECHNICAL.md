# SAM Coupé Screen Modes & Palette — Technical Reference

*Based on the implementation in the SAM Coupé SCREEN$ Editor, verified against the SAMDOS source,
SimCoupe emulator source, and scrimage/palette.py.*

---

## Palette

The SAM Coupé has a **128-colour palette** encoded in a 7-bit index with the following bit layout:

```
Bit: 6    5    4    3    2    1    0
     GRN1 RED1 BLU1 BRGT GRN0 RED0 BLU0
```

Each channel (R, G, B) uses a **3-bit index** into an 8-entry intensity table:

```
intensity = [0x00, 0x24, 0x49, 0x6d, 0x92, 0xb6, 0xdb, 0xff]
           = [  0,   36,   73,  109,  146,  182,  219,  255]
```

The 3-bit channel indices are formed as:

```python
r = (i & 0x02)      | ((i & 0x20) >> 3) | ((i & 0x08) >> 3)
g = (i & 0x04) >> 1 | ((i & 0x40) >> 4) | ((i & 0x08) >> 3)
b = (i & 0x01) << 1 | ((i & 0x10) >> 2) | ((i & 0x08) >> 3)
```

The BRIGHT bit (bit 3) contributes **+1** to each channel's 3-bit index — it is not a flat offset
but a step up the intensity table. This means BRIGHT on a zero channel gives 36 (not 0), and
BRIGHT on a fully-saturated channel gives 255 (capped at table top).

This is an exact port of `get_sam_palette()` from scrimage/palette.py, which produces colours
matching SimCoupe's output.

## SCREEN$ File Format

All modes share a common palette/interrupt block appended after the pixel data:

```
+0      : 16 bytes — Mode 4 Palette A (CLUT slots 0-15, palette indices)
+16     : 4 bytes  — Mode 3 store (HMPR register state)
+20     : 16 bytes — Mode 4 Palette B (duplicate of A in most cases)
+36     : 4 bytes  — Mode 3 store B
+40     : Line interrupt records (4 bytes each), terminated by 0xFF
```

Each line interrupt record:

```
Byte 0: Y line (0-191) — stored as y, fires at scan line y+1
Byte 1: CLUT slot (0-based)
Byte 2: New palette index for slot (Palette A)
Byte 3: New palette index for slot (Palette B)
```

Maximum **127 line interrupt records** (SAM hardware limit).

The SAMDOS directory entry stores the screen mode at byte 221 as a **0-based** value:
`0=Mode 1, 1=Mode 2, 2=Mode 3, 3=Mode 4`

## Mode 1 — Spectrum Compatible (256×192, 1bpp + attributes)

**Total size**: 6144 + 768 + palette block = 6912 + palette

**Pixel data** (6144 bytes): ZX Spectrum non-linear framebuffer layout:

```
Address bits: [15:14]=01, [13:11]=row_in_third, [10:8]=third, [7:5]=char_row, [4:0]=col
```

Each byte encodes 8 horizontal pixels, MSB = leftmost. Pixel value 0 = paper, 1 = ink.

**Attribute data** (768 bytes, at offset 6144): 32×24 cells (8×8 pixels each):

```
Bit 7   : Flash
Bit 6   : Bright
Bits 5-3: Paper colour (0-7, maps to SAM CLUT)
Bits 2-0: Ink colour (0-7, maps to SAM CLUT)
```

---

## Mode 2 — Enhanced Attribute (256×192, 1bpp + 8×1 attributes)

**Total size**: 6144 + 6144 + palette block = 14336 + palette (+ 2048 bytes padding between
pixel and attr blocks)

**Pixel data** (6144 bytes at offset 0): Linear — row 0 first, 32 bytes per row,
1 bit per pixel, MSB = leftmost.

**Attribute data** (6144 bytes at offset 8192): 32×192 cells — one attribute byte per
8×1 pixel strip. Same byte format as Mode 1 (BRIGHT, Paper 3-bit, Ink 3-bit). This gives
per-scanline colour control without line interrupts.

---

## Mode 3 — High Resolution (512×192, 2bpp)

**Total size**: 24576 + palette block

**Pixel data** (24576 bytes): Linear, 512 pixels wide × 192 rows. Each byte stores 4 pixels
packed as 2-bit pairs. **Bit reversal**: each 2-bit pair is stored with bits reversed:

```
stored_pair = ((pixel & 1) << 1) | (pixel >> 1)
```

Applied on both import and export. 4 colours per scan line, selected from CLUT slots 0-3.

---

## Mode 4 — Standard (256×192, 4bpp)

**Total size**: 24576 + palette block

**Pixel data** (24576 bytes): Linear, 256 pixels wide × 192 rows. Each byte stores 2 pixels
as two 4-bit nibbles:

```
byte = (left_pixel << 4) | right_pixel
```

16 colours per scan line, selected from CLUT slots 0-15.

## Line Interrupts (all modes)

Line interrupts allow CLUT slot values to change mid-screen, giving more than 16 simultaneous
colours. The hardware fires each interrupt at the **end** of the stored scan line, making the
new colour visible from the **next** line. Therefore:

- To change a colour starting at scan line Y: store `line = Y - 1`
- Rendering applies records where `record.line === currentY - 1`

Up to **127 records** fit in the SCREEN$ palette block. Records are sorted by line number
before rendering. When line interrupts are generated by image import optimisation, only records
that introduce a colour genuinely different from all current CLUT entries are emitted (controlled
by the minimum improvement threshold).

## MGT Disk Image Format

**File size**: exactly 819,200 bytes (80 tracks × 2 sides × 10 sectors × 512 bytes)

**Physical layout**: sides are **interleaved per track**:

```
[track0_side0][track0_side1][track1_side0][track1_side1]...[track79_side0][track79_side1]
```

**Sector offset formula**:

```javascript
offset = (track * 2 * 10 + side * 10 + (sector - 1)) * 512
```

**Chain addressing**: track 0–79 = side 0, track 128–207 = side 1.

**Directory**: tracks 0–3 (40 sectors × 2 entries = 80 slots). Each entry is 256 bytes:

```
Byte 0      : file type (low 5 bits); type 20 = SCREEN$
Bytes 1-10  : filename (space-padded)
Byte 11     : sector count MSB
Byte 12     : sector count LSB
Byte 13     : start track
Byte 14     : start sector
Bytes 15-209: sector address map (BAM)
Byte 221    : screen mode (0-based: 0=M1, 1=M2, 2=M3, 3=M4)
Byte 237    : modulo LSB
Byte 238    : modulo MSB
Byte 239    : number of 16KB pages
```

**File header** (first 9 bytes of file data on disk):

```
Byte 0  : file type
Bytes 1-2: modulo (file length mod 16384), LSB first
Byte 7  : number of 16KB pages
```

Actual file length = pages × 16384 + modulo

**Sector chain**: bytes 510–511 of each 512-byte sector = [next_track, next_sector].
Chain ends when both are 0.

## CLUT (Colour Look-Up Table)

All modes use a CLUT to map pixel values to SAM palette indices:

- **Modes 1 & 2**: 8-entry CLUT (ink/paper values 0–7 map to palette indices)
- **Mode 3**: 4-entry CLUT (pixel values 0–3)
- **Mode 4**: 16-entry CLUT (pixel values 0–15)

The CLUT is stored in the palette block at the end of the SCREEN$ file (first 16 bytes).
Line interrupts modify CLUT entries mid-frame, allowing more colours than the slot count
would normally permit.

## File Extensions

| Extension | Mode |
|-----------|------|
| `.ss1`    | Mode 1 |
| `.ss2`    | Mode 2 |
| `.ss3`    | Mode 3 |
| `.ss4`    | Mode 4 |

Files saved to MGT disk images by SAMDOS are type 20 (SCREEN$). Files transferred via
SAMdisk or similar tools may be saved as type 19 (CODE) and require manual mode selection
when loading.
